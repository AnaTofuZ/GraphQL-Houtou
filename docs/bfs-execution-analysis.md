# 幅優先(BFS)実行の効果検討 (2026-07-15)

「実行を幅優先(セレクションツリーのレベル順)にすると DataLoader の
バッチングが改善するか」の検討結果。**結論: 導入効果なし。現行の
stall ベース実行は「依存深度バッチング」であり、レベル順 BFS の上位互換
になっている。**

## 背景: なぜ BFS が話題になるか

素朴な深さ優先実行の executor(resolver を await しながら木を降りる)では、
兄弟フィールドのロードが直列化されて N+1 が悪化する。これを避ける定石が

- graphql-js: 深さ優先だが promise を待たずに兄弟へ進み、イベントループの
  tick 単位で DataLoader が flush される(実質「その tick に溜まったもの」
  単位のバッチング)
- 一部の実装(gqlgen の dataloaden 運用など): レベル順 BFS で「深さ N の
  ロードをまとめて発行」する設計が紹介される

Houtou は promise を待たずに実行を進め、**どこにも進めなくなった時点
(stall)で on_stall が全 loader を flush する**。バッチの単位は
「ツリーのレベル」ではなく**「新しいデータなしに到達できる範囲」= 依存の
波(wave)**になる。

## 比較: レベル順 BFS vs 依存深度(stall)バッチング

ロードをまとめられる上限は「そのロードの入力が揃った時点」で決まる。

- **レベル順 BFS**: 深さ d のロードは深さ d の処理時に発行される。
  入力が既に揃っていても、浅い階層のロードとはまとめられない。
- **stall ベース**: 同期的に降りられる部分(resolver が即値を返す中間
  オブジェクト)はすべて即時に降り切るので、**木の深さが違っても依存が
  ないロードは同じ波に入る**。

つまり stall ベースのバッチ数 ≤ レベル順 BFS のバッチ数(バッチは
loader ごと・波ごとに 1 回)。逆方向の差が付くケースは存在しない:
stall はまさに「これ以上どのロードも発行できない」まで待つ、最も遅延した
flush だから。

## 実測

形状: `shallowUser`(木の深さ 1 で users.load)と
`deep.inner.user`(中間 2 階層は即値オブジェクト、深さ 3 で users.load)、
さらに `user.boss`(深さ 4、ロード済みキーの再ロード)。

```
{ shallowUser { name } deep { inner { user { name boss { name } } } } }
```

- **現行(stall ベース)**: users のバッチは **1 回**(`u2,u1` が同じ波)。
  boss はリクエスト内キャッシュにヒットし追加バッチなし
- **レベル順 BFS(理論値)**: 深さ 1 で `u2`、深さ 3 で `u1` → **2 回**
  (+ boss がキャッシュされないなら 3 回)

同一依存深度の多段形状(posts → author → team)は従来から
「1 波 = 1 loader バッチ」で、t/36 が回帰テストとして固定済み。
今回の異深度形状も t/41 に回帰テストを追加した。

## BFS が意味を持ちうる残りの論点

1. **first-flush のレイテンシ**: BFS なら浅いロードが「深い部分の同期
   実行を待たずに」発行できる、という議論。単一プロセスで resolver が
   同期実行される Perl では、flush の発行順が入れ替わるだけで総所要時間は
   変わらない(I/O を非同期に多重化しないため)。イベントループ統合
   (loader の flush を I/O 待ちと重ねる)を導入する場合に再検討すべき
   論点であり、実行順の BFS 化そのものには効果がない。
2. **メモリ**: BFS はレベル全体の中間状態を保持する。stall ベースは
   pending entry / frame として同等の情報を保持しており差はない。
3. **serial mutation**: 仕様上 root フィールドを直列実行するため、
   どちらの戦略でもバッチングは root フィールド単位に閉じる(現行どおり)。

## 結論

- 実行順を幅優先に変える改修は行わない。stall ベースの依存深度
  バッチングが、バッチ回数の意味でレベル順 BFS の上位互換であることを
  実測で確認した(異深度 1 バッチの回帰テストを t/41 に固定)
- 将来イベントループ統合を検討する際は、本書 1. の first-flush 論点を
  再評価すること
