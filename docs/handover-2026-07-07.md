## ハマりどころ(今セッションで 2 回踏んだものを含む)

- **`perl Build` はヘッダ(.h)のみの変更で .o を再コンパイルしない**。
  vm_runtime.h や src/*.h を直したら
  `rm -f lib/GraphQL/Houtou.o lib/GraphQL/Houtou.c && perl Build`。
  「修正したのに直らない」「壊してないのに壊れた」の大半はこれ。
  `minil test` は毎回クリーンビルドなので最終確認はそちら
- **minilla は git 未追跡ファイルをテスト/パッケージに含めない**。
  新規 t/*.t は `git add` してから `minil test`(Files= の数で確認)
- **Perl 側アリーナ census(walk_arena diff)はスナップショット間で
  SV を大量に確保するとアドレス再利用で偽陰性が出る**。
  リクエスト内部の誕生時刻を絞るときは C 側で SV を確保しない
  アリーナ走査スナップ(current-context.md の 2026-07-07 L1 節参照)を
  実行パスに挟み、二分探索する。`Devel::FindRef` / `Devel::MAT` は
  C 構造体が保持する SV の所有者を辿れない(「not found」になる)
- newAV 起源の特定には `#define newAV()` トレースマクロが有効
  (bootstrap.h 直後に置けば全ヘッダをカバーできる)。
  Perl コア inline 生成(`[]` 等)は捕まらないので
  `Perl_av_make` への lldb ブレークと組み合わせる

## async lane の要点(コード読解の起点)

- 実行系統: variables なし/`on_stall` あり →
  `execute_native_program_auto_*`(exec-state lane、promise 対応)。
  variables あり + `on_stall` なし → `execute_native_program_handle_xs`
  (fast lane、promise 非対応 = #28)
- `on_stall` 契約: executor がストール(ready queue 空 + pending 残)する
  たびに呼ばれ「進捗数」を返す。進捗 0 + pending 残 = deadlock で die。
  駆動は `NativeRuntime::_settle_result`(Promise::XS の resolve は同期に
  コールバックが走る)
- `s->response_frame`: リクエスト最初の block frame。この frame の
  deferred だけが response envelope を resolve する
  (`deferred_resolves_response`)。`frame_stack_count == 1` 判定は
  late continuation で誤るため廃止済み
- L2: `s->response_json_mode` が立っていると response frame の resolve 時に
  `gql_runtime_vm_response_json_from_native_sv`(native tree walker)で
  JSON を直接生成。全 sync 完了時は `response_json_from_data_sv`(SV walker)
- 所有権規約: `resolve_deferred_xs` は値の所有権を取らない
  (呼び出し側が dec)。`expect_error_records_av` は borrowed AV か
  NULL を返す(以前は owned な newAV を返してリークしていた)
